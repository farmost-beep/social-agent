# 社交关系专家系统 设计规约 (SPEC)

> 本规约定义系统的设计理念、数据模型、操作规则和演进原则。
> 所有对该系统的修改、查询、扩展均须遵循本文档。

---

## 0. 理论基础

### 0.1 关系强度的本质

社交关系强度不是对一个人"社会地位"或"有用程度"的打分，而是对**"你跟这个人之间那条连接的深度"**的度量。

强度高 ≠ 这个人"重要"（在功利意义上）
强度高 ＝ 你们之间的**连接深厚**（在关系意义上）

一个快递员可以强度5（家人），一个行长可以强度4（老领导），一个CEO可以强度1（仅认识）。**强度衡量的不是对方，而是"你和他之间"。**

### 0.2 三层决定因素

```
强度 = f(血缘纽带, 生涯纽带, 场景纽带)
```

| 层次 | 决定因素 | 典型强度 | 举例 |
|:----:|---------|:--------:|------|
| **血缘纽带** | 不可替代的亲密关系 | **5** | 配偶、子女、父母、挚友 |
| **生涯纽带** | 对人生轨迹有重要影响的人 | **4** | 工作领导、导师、引路人 |
| **场景纽带** | 因特定场景产生的连接 | **2-3** | 同事、校友、社团成员 |

### 0.3 核心原则

**原则一：关系优先于身份。** 系统回答的是"我怎么认识他的"，而不是"他是干什么的"。一个人的职务可以变，但他和你的连接方式不会变。

**原则二：强度不可批量设定。** 每段关系的强度必须由用户逐一确认，不能根据公司名、学校名等标签批量赋值。上一轮批量的邮储同事升3已被纠正。

**原则三：强度≠有用性。** 系统的目的是帮用户**经营关系**，不是帮用户**利用资源**。冷却检查（14天预警、21天告警）体现的就是"关系需要持续经营"的理念。

**原则四：关系是动态的。** 强度不是静态标签，会随着时间推移和互动频率自然变化。长期不联系的关系会自动冷却提醒。

## 1. 核心设计理念

### 1.1 以人+关系为核心

系统不是通讯录，而是**关系网**。每个联系人的核心价值不是他的职务有多高，而是**他跟用户的连接是什么**。

- 记录一个人时，优先回答：**"我怎么认识他的？"** 而非 "他是干什么的？"
- 关系强度（strength）必须反映**用户与该人的实际关系深浅**，而非根据单位/职务批量设定
- 家人至亲应设最高强度（5），无论他们是否在职场上"重要"

### 1.2 统一数据源

**所有社交关系只记录在此系统内**，不在 Claude Code 的通用记忆系统（memory/）或其他地方重复存储。

| ✅ 正确做法 | ❌ 错误做法 |
|-----------|-----------|
| 社交关系 → social-agent/data/contacts.json | 社交关系 → memory/*.md |
| 用 `note` 命令加结构化记忆 | 用通用记忆文件存社交信息 |

### 1.3 社交资源的定义

"社交相关资源"、"社交关系"、"认识的人"、"校友资源"、"人脉"等，指的都是**记入此系统的关系**，而非泛指所有人。只有经过筛选、评估、结构化记录的社交关系才算"社交资源"。

## 2. 数据模型

### 2.1 联系人 (Contact)

```json
{
  "id": "唯一标识符",
  "name": "显示名称",
  "relation": "关系分类 (self/family/校友/同行/创业/合作/其他)",
  "sub_relation": "子分类 (如：邮储银行总行/科大校友/金融同行)",
  "strength": 1-5,        // 关系强度
  "tags": ["标签1", "标签2"],
  "platforms": {
    "weixin": "微信号",
    "phone": "手机号",
    "email": "邮箱"
  },
  "notes": "自由文本备注",
  "memories": [],          // 结构化记忆数组
  "important_dates": [],   // 重要日期数组
  "stage": "",             // 关系阶段
  "created": "创建日期"
}
```

### 2.2 强度 (strength) 定义

| 强度 | 含义 | 示例 |
|:----:|------|------|
| **5** | **至亲** | 自己、配偶、子女、父母、挚友 |
| **4** | **重要** | 工作履历上的**所有领导**、深度合作/师徒/长期老友/重要引路人 |
| **3** | **较熟** | 同事、有交集的校友、常联系的朋友 |
| **2** | **普通** | 通讯录基础关系，认识但不常联系 |
| **1** | **弱关系** | 仅存联系方式，几乎无互动 |

**重要规则：** 强度不应批量设定。每个人的强度应基于**用户与该人的实际关系**逐一评估。

### 2.3 记忆 (Memory)

结构化信息条目，每条独立记录并附带标签：

```json
{
  "id": "mem-xxxxx",
  "content": "记忆内容",
  "tags": ["标签"],
  "created": "创建时间"
}
```

使用场景：记录履历、共同经历、重要事件、个人特征等。避免把所有信息塞进 notes 字段。

### 2.4 时间线 (Timeline)

互动记录，每次与联系人的交流、事件都应记入时间线：

```json
{
  "id": "t-xxxxx",
  "date": "日期",
  "contact": "联系人ID",
  "type": "类型 (message/meeting/call/milestone)",
  "summary": "摘要",
  "pending": "待办事项",
  "key_points": ["关键点"]
}
```

## 3. 操作规则

### 3.1 查询规则

涉及联系人数据的查询，**禁止从记忆或上下文猜答案**，必须读 `data/contacts.json` 获取真实数据。

### 3.2 修改规则

- 新增联系人 → `add-contact` 命令
- 更新联系人 → `edit-contact` 命令
- 添加记忆 → `note` 命令（逐条添加，不堆notes里）
- 记录互动 → `log` 命令
- 数据修改必要时用 Python 脚本直接操作 JSON

### 3.3 关系维护规则

- 每次添加记忆（note）时，如果关系强度偏低（<5），自动+1
- 超过**14天**未联系的关系标记为🟡冷却
- 超过**21天**未联系的关系标记为🔴预警
- 生日检测从 notes 中自动提取（如"1129"→11月29日）

## 6. 演进原则

1. **增量演进**：不追求一次性完美，每次交互中持续优化
2. **数据驱动**：所有设计决策基于实际使用反馈，而非预设
3. **人本优先**：任何技术优化不能削弱"以人+关系为核心"的设计理念
4. **最小冗余**：一条信息只在一个地方存储，不跨系统重复
5. **查询优先**：优化读端体验（微信端），写端可接受适当延迟

## 7. 自动化规则

### 7.1 基于时间线的强度动态调整

系统利用时间线中的互动记录，定期分析关系强度是否需要调整。

**降级触发（长期无互动）：**

| 当前强度 | 90天无互动 | 180天无互动 |
|:--------:|:----------:|:-----------:|
| **4**（领导/引路人） | 建议降为3 | 建议降为2 |
| **3**（较熟） | 建议降为2 | 建议降为1 |

**升级触发（高频互动）：**
- 近**60天内≥2次**互动，且当前强度<4 → 建议升1级（不超过4）

**保护规则：**
- 强度5（家人）：**永不自动调整**
- 强度2及以下：**永不自动调整**
- self（自己）：**永不自动调整**
- 升降冲突时：**降级优先**

**执行方式：**
- `social.py adjust` → 查看建议
- `social.py adjust --apply` → 确认执行

### 7.2 记忆备注的轻度提升

添加记忆（`note`命令）时自动提升强度1级，但不超过4，不碰5级家人。
