# 社交关系专家系统 设计规约 (SPEC)

> 本规约定义系统的设计理念、数据模型、操作规则和演进原则。
> 所有对该系统的修改、查询、扩展均须遵循本文档。

---

## 0. 理论基础

### 0.1 关系强度的定义：社交价值

社交关系强度按**社交价值**排序——即这个人对用户**实现幸福人生总体目标**的贡献度。

幸福人生六维目标框架：
1. **投资成功** — 长期财务目标
2. **事业进阶** — 科技金融投资框架成熟
3. **家庭支持** — 子女成才、夫妻和谐、父母安康、财务稳健
4. **百岁健康** — 运动习惯、指标达标
5. **AI能力** — 独立开发自动化工具
6. **知识库** — 系统化可检索的知识体系

社交价值的大小，取决于这个人能帮助撬动六维目标中的哪一个、撬动多大。

| 强度 | 社交价值等级 | 典型人物 |
|:----:|:-----------|:--------|
| **5** | **能直接影响决策 + 家人** | 直接上级、关键决策者、配偶、子女 |
| **4** | **能深度合作/提供重要机会** | 资深同行、创业伙伴、重要前领导 |
| **3** | **有价值的信息/资源交换对象** | 同业校友、行业专家、合作方 |
| **2** | **普通社交关系** | 一般校友、泛泛之交 |
| **1** | **弱关系** | 仅存联系方式，几乎无互动 |

### 0.2 社会角色框架

社交关系的本质是**社会角色与角色之间的互动**。每个联系人不是单一的人，而是以某个角色身份与你互动。

你的社会角色体系：

| 层级 | 角色 | 当前身份 |
|:----:|:----|:--------|
| **一级** | 🏦 邮储银行科技金融高管 | 上海分行科技金融事业部高级副经理 |
| | 🏛️ 前华瑞银行审计负责人 | 前审计部总经理/监事会办公室主任 |
| | 💼 前外资银行内审专家 | 前东亚银行内审、前大都会内审 |
| **二级** | 🎓 中科大人 | 9509精密机械+计算机双学位 |
| | 🤝 民建会员 | 中国民主建国会 |
| | ✍️ 金融科技研究者 | 14条阅读主线、475本馆藏 |
| **三级** | 👨‍👩‍👦 家庭支柱 | 丈夫、父亲、儿子 |
| | 💰 个人投资者 | 目标长期财务目标 |
| | 🤖 AI实践者 | Claude Code日常使用 |

**社交价值公式：**

```
社交价值 = Σ (我的角色A × 对方角色B × 双向互动力)
```

- 角色的匹配度决定了**互动的可能性**（邮储高管×邮储总行领导 = 同体系，互动成本低）
- 双向互动力决定了**互动的可实现性**（某企业高层潜在价值5，但双向互动力不足，实际只到4）
- 强度应反映**在特定角色关系下已实现的互动价值**

### 0.3 角色互动层数

社交关系质量由**角色互动层数**决定——即双方在多少个不同社会角色维度上存在交集。

**角色维度定义：**

| 维度 | 标签 | 含义 |
|:----:|:----:|:----|
| **职业层** | `职层` | 当前/过去在同一单位或行业 |
| **校友层** | `校层` | 中科大校友 |
| **组织层** | `组层` | 民建等同组织成员 |
| **社交层** | `群层` | 同在某个微信/社交群 |
| **合作层** | `合层` | 有项目合作关系 |
| **家庭层** | `家层` | 配偶、子女、父母等家人 |

**层数越高，社交交集越密：**

| 层数 | 含义 | 系统标记 | 示例 |
|:---:|:----|:--------|:----|
| **3层** | 三重角色叠加 | `角色x3` | 某人（校友+金融同行+群友） |
| **2层** | 双重角色叠加 | `角色x2` | 某人（现公司领导+校友） |
| **1层** | 单层角色 | `角色x1` | 家人、纯校友、纯群友 |

角色互动层数作为辅助参考，不直接决定强度，但与社交价值正相关。

### 0.4 核心原则

**原则一：社交价值排序。** 强度高 = 这个人在你的事业/业务/信息网络中位置更重要。不按情感亲疏排。

**原则二：强度不可批量设定。** 每段关系的强度必须由用户逐一确认，不能根据公司名、学校名等标签批量赋值。

**原则三：强度随社交价值变化而变。** 这个人换工作、升职、或你们的合作深化了，强度就应该调整。

**原则四：潜在价值≠实际价值。** 社交价值需要**双向互动**才能兑现。一个人理论上能帮你（潜在价值5），但如果没有实际互动和关系深度（双向互动），实际社交价值不应高于4。强度应反映**已实现的社交价值**，而非理论上的上限。

**原则五：关系需要持续经营。** 即使社交价值高的人，长期不互动也会冷却。冷却检查（14天预警、21天告警）体现的就是"价值关系需要维护"。

## 1. 核心设计理念

### 1.1 以人+关系为核心

系统不是通讯录，而是**关系网**。每个联系人的核心价值不是他的职务有多高，而是**他跟用户的连接是什么**。

- 记录一个人时，优先回答：**"我怎么认识他的？"** 而非 "他是干什么的？"
- 关系强度（strength）必须反映**用户与该人的实际关系深浅**，而非根据单位/职务批量设定
- 强度按社交价值排序，不按情感亲疏

### 1.2 统一数据源

**所有社交关系只记录在此系统内**，不在 Claude Code 的通用记忆系统（memory/）或其他地方重复存储。

| ✅ 正确做法 | ❌ 错误做法 |
|-----------|-----------|
| 社交关系 → social-agent/data/contacts.json | 社交关系 → memory/*.md |
| 用 `note` 命令加结构化记忆 | 用通用记忆文件存社交信息 |

### 1.3 社交资源的定义

"社交相关资源"、"社交关系"、"认识的人"、"校友资源"、"人脉"等，指的都是**记入此系统的关系**，而非泛指所有人。只有经过筛选、评估、结构化记录的社交关系才算"社交资源"。

## 2. 数据模型

### 2.1 联系人 (Contact)

```json
{
  "id": "唯一标识符",
  "name": "显示名称",
  "relation": "关系分类 (self/family/校友/同行/创业/合作/其他)",
  "sub_relation": "子分类 (如：邮储银行总行/科大校友/金融同行)",
  "strength": 1-5,        // 关系强度
  "tags": ["标签1", "标签2"],
  "platforms": {
    "weixin": "微信号",
    "phone": "手机号",
    "email": "邮箱"
  },
  "notes": "自由文本备注",
  "memories": [],          // 结构化记忆数组
  "important_dates": [],   // 重要日期数组
  "stage": "",             // 关系阶段
  "created": "创建日期"
}
```

### 2.2 强度 (strength) 定义

| 强度 | 社交价值 | 示例 |
|:----:|:--------|------|
| **5** | **直接影响决策** | 直接上级、关键决策者、重要客户 |
| **4** | **重要合作/引路人** | 前领导、创业伙伴、资深同行、深度合作方 |
| **3** | **有价值的信息交换** | 同业校友、行业专家、潜在合作方 |
| **2** | **普通社交关系** | 一般校友、泛泛之交 |
| **1** | **弱关系** | 仅存联系方式，几乎无互动 |

**重要规则：** 强度不应批量设定。每个人的强度应基于其对用户的**社交价值**逐一评估。

### 2.3 记忆 (Memory)

结构化信息条目，每条独立记录并附带标签：

```json
{
  "id": "mem-xxxxx",
  "content": "记忆内容",
  "tags": ["标签"],
  "created": "创建时间"
}
```

使用场景：记录履历、共同经历、重要事件、个人特征等。避免把所有信息塞进 notes 字段。

### 2.4 时间线 (Timeline)

互动记录，每次与联系人的交流、事件都应记入时间线：

```json
{
  "id": "t-xxxxx",
  "date": "日期",
  "contact": "联系人ID",
  "type": "类型 (message/meeting/call/milestone)",
  "summary": "摘要",
  "pending": "待办事项",
  "key_points": ["关键点"]
}
```

## 3. 操作规则

### 3.1 查询规则

涉及联系人数据的查询，**禁止从记忆或上下文猜答案**，必须读 `data/contacts.json` 获取真实数据。

### 3.2 修改规则

- 新增联系人 → `add-contact` 命令
- 更新联系人 → `edit-contact` 命令
- 添加记忆 → `note` 命令（逐条添加，不堆notes里）
- 记录互动 → `log` 命令
- 数据修改必要时用 Python 脚本直接操作 JSON

### 3.3 关系维护规则

- 每次添加记忆（note）时，如果关系强度偏低（<5），自动+1
- 超过**14天**未联系的关系标记为🟡冷却
- 超过**21天**未联系的关系标记为🔴预警
- 生日检测从 notes 中自动提取（如"1129"→11月29日）

## 6. 演进原则

1. **增量演进**：不追求一次性完美，每次交互中持续优化
2. **数据驱动**：所有设计决策基于实际使用反馈，而非预设
3. **人本优先**：任何技术优化不能削弱"以人+关系为核心"的设计理念
4. **最小冗余**：一条信息只在一个地方存储，不跨系统重复
5. **查询优先**：优化读端体验（微信端），写端可接受适当延迟

## 7. 自动化规则

### 7.1 基于时间线的强度动态调整

系统利用时间线中的互动记录，定期分析关系强度是否需要调整。

**降级触发（长期无互动）：**

| 当前强度 | 90天无互动 | 180天无互动 |
|:--------:|:----------:|:-----------:|
| **4**（领导/引路人） | 建议降为3 | 建议降为2 |
| **3**（较熟） | 建议降为2 | 建议降为1 |

**升级触发（高频互动）：**
- 近**60天内≥2次**互动，且当前强度<4 → 建议升1级（不超过4）

**保护规则：**
- 强度5（决策层）：**永不自动调整**（社交价值最高级，不可自动降）
- 强度2及以下：**永不自动调整**
- self（自己）：**永不自动调整**

**执行方式：**
- `social.py adjust` → 查看建议
- `social.py adjust --apply` → 确认执行

### 7.2 记忆备注的轻度提升

添加记忆（`note`命令）时自动提升强度1级，但不超过4，不碰5级家人。
