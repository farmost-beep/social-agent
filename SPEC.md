# 社交关系专家系统 设计规约 (SPEC)

> 本规约定义系统的设计理念、数据模型、操作规则和演进原则。
> 所有对该系统的修改、查询、扩展均须遵循本文档。

---

## 1. 核心设计理念

### 1.1 以人+关系为核心

系统不是通讯录，而是**关系网**。每个联系人的核心价值不是他的职务有多高，而是**他跟用户的连接是什么**。

- 记录一个人时，优先回答：**"我怎么认识他的？"** 而非 "他是干什么的？"
- 关系强度（strength）必须反映**用户与该人的实际关系深浅**，而非根据单位/职务批量设定
- 家人至亲应设最高强度（5），无论他们是否在职场上"重要"

### 1.2 统一数据源

**所有社交关系只记录在此系统内**，不在 Claude Code 的通用记忆系统（memory/）或其他地方重复存储。

| ✅ 正确做法 | ❌ 错误做法 |
|-----------|-----------|
| 社交关系 → social-agent/data/contacts.json | 社交关系 → memory/*.md |
| 用 `note` 命令加结构化记忆 | 用通用记忆文件存社交信息 |

### 1.3 社交资源的定义

"社交相关资源"、"社交关系"、"认识的人"、"校友资源"、"人脉"等，指的都是**记入此系统的关系**，而非泛指所有人。只有经过筛选、评估、结构化记录的社交关系才算"社交资源"。

## 2. 数据模型

### 2.1 联系人 (Contact)

```json
{
  "id": "唯一标识符",
  "name": "显示名称",
  "relation": "关系分类 (self/family/校友/同行/创业/合作/其他)",
  "sub_relation": "子分类 (如：邮储银行总行/科大校友/金融同行)",
  "strength": 1-5,        // 关系强度
  "tags": ["标签1", "标签2"],
  "platforms": {
    "weixin": "微信号",
    "phone": "手机号",
    "email": "邮箱"
  },
  "notes": "自由文本备注",
  "memories": [],          // 结构化记忆数组
  "important_dates": [],   // 重要日期数组
  "stage": "",             // 关系阶段
  "created": "创建日期"
}
```

### 2.2 强度 (strength) 定义

| 强度 | 含义 | 示例 |
|:----:|------|------|
| **5** | **至亲** | 自己、配偶、子女、父母、挚友 |
| **4** | **重要** | 深度合作/师徒/长期老友/重要引路人 |
| **3** | **较熟** | 同事、有交集的校友、常联系的朋友 |
| **2** | **普通** | 通讯录基础关系，认识但不常联系 |
| **1** | **弱关系** | 仅存联系方式，几乎无互动 |

**重要规则：** 强度不应批量设定。每个人的强度应基于**用户与该人的实际关系**逐一评估。

### 2.3 记忆 (Memory)

结构化信息条目，每条独立记录并附带标签：

```json
{
  "id": "mem-xxxxx",
  "content": "记忆内容",
  "tags": ["标签"],
  "created": "创建时间"
}
```

使用场景：记录履历、共同经历、重要事件、个人特征等。避免把所有信息塞进 notes 字段。

### 2.4 时间线 (Timeline)

互动记录，每次与联系人的交流、事件都应记入时间线：

```json
{
  "id": "t-xxxxx",
  "date": "日期",
  "contact": "联系人ID",
  "type": "类型 (message/meeting/call/milestone)",
  "summary": "摘要",
  "pending": "待办事项",
  "key_points": ["关键点"]
}
```

## 3. 操作规则

### 3.1 查询规则

涉及联系人数据的查询，**禁止从记忆或上下文猜答案**，必须读 `data/contacts.json` 获取真实数据。

### 3.2 修改规则

- 新增联系人 → `add-contact` 命令
- 更新联系人 → `edit-contact` 命令
- 添加记忆 → `note` 命令（逐条添加，不堆notes里）
- 记录互动 → `log` 命令
- 数据修改必要时用 Python 脚本直接操作 JSON

### 3.3 关系维护规则

- 每次添加记忆（note）时，如果关系强度偏低（<5），自动+1
- 超过**14天**未联系的关系标记为🟡冷却
- 超过**21天**未联系的关系标记为🔴预警
- 生日检测从 notes 中自动提取（如"1129"→11月29日）

## 6. 演进原则

1. **增量演进**：不追求一次性完美，每次交互中持续优化
2. **数据驱动**：所有设计决策基于实际使用反馈，而非预设
3. **人本优先**：任何技术优化不能削弱"以人+关系为核心"的设计理念
4. **最小冗余**：一条信息只在一个地方存储，不跨系统重复
5. **查询优先**：优化读端体验（微信端），写端可接受适当延迟
