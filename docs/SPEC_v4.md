## §16. 初心与诊断

### 16.1 产品初心（三个承诺）

产品的原始承诺（SKILL.md 第一行）：

> 自动追踪所有社交关系中「**最近干了什么**」和「**下一步该干什么**」。

结合 SPEC §0 的社交价值理论，初心拆解为三个承诺：

| # | 承诺 | 含义 |
|:-:|:----|:----|
| ① | **记得住** | 最近干了什么（timeline） |
| ② | **想得到** | 下一步该干什么（主动建议） |
| ③ | **对得上** | 每一步经营服务于用户的人生目标（社交价值排序） |

### 16.2 v3.0 时点体检（2026-07-10 实测）

| 承诺 | 现状 | 兑现度 |
|:----|:----|:---:|
| ① 记得住 | timeline 可用（32条），手动"记一下"摩擦已低 | 🟢 ~70% |
| ② 想得到 | 仅有倒计时式提醒（14/21天冷却、health分、生日），回答"谁凉了"而非"该干什么" | 🟡 ~30% |
| ③ 对得上 | §0.1 六维目标框架无任何数据字段承接，4,683人中0人标注目标撬动关系 | 🔴 0% |

**核心诊断**：系统现在是一支**关系温度计**——精确告诉你哪段关系变凉了。初心要的是一位**经营参谋**——告诉你"联系谁、为什么、聊什么"。温度计回答 when，参谋回答 why 和 what。

**广度路线证伪**（实测数据）：v2.5 假设"补全3,012个灰色联系人→变成资产"，管道就绪一个月仅处理3人（3/4,683）。实际经营半径是 timeline 上的 ~30 人活跃圈。结论：**下一版走深耕路线，不走广度路线。**

## §18. v4.0 支柱一：目标锚定（承诺③落地）

### 18.1 目标框架

采用用户真实的六维目标框架（✅ 已确认 2026-07-10）：

1. **事业** — 职业晋升、业务拓展、方法论落地
2. **投资** — 财富增长、组合优化、投资能力
3. **家庭** — 子女成长、夫妻关系、父母赡养
4. **健康** — 运动习惯、体检指标、身心状态
5. **AI能力** — AI工具链、自动化、认知升级
6. **知识** — 阅读体系、技能习得、知识库建设

框架存于 `config/goals.yaml`，用户可自定义。

### 18.2 数据模型：leverage 字段

联系人（仅 core 层）新增：

```json
"leverage": {
  "goals": ["事业", "AI能力"],
  "how": "民建人脉+科技金融业务引荐",
  "direction": "互惠",
  "confirmed": "2026-07-15"
}
```

| 字段 | 类型 | 说明 |
|:----|:----|:----|
| `goals` | string[] | 该联系人撬动的目标维度（可多个） |
| `how` | str | 具体撬动方式，一句话 |
| `direction` | enum | `我求于他` / `他求于我` / `互惠` |
| `confirmed` | date | 用户确认日期（空=仅AI建议，未生效） |

### 18.3 锚定规则

- **遵守原则二（不可批量设定）**：AI 逐人生成锚定建议，用户逐人确认后 `confirmed` 才生效
- **仅 core 层锚定**：不给储备池4,366人做锚定——参谋只参谋棋盘上的子
- **节奏（✅ 已确认 2026-07-10）**：先锚定强度4-5的31人快速见效，其后每批10人过强度3的286人
- 新命令：
```bash
social anchor              # 交互式锚定：AI建议→用户确认，每次一批
social anchor <contact>    # 单人锚定
social anchor --stats      # 锚定进度统计
```

### 18.4 实施记录（2026-07-10 阶段2落地）

| 项目 | 决策 | 理由 |
|:----|:----|:----|
| `leverage` 存储位置 | 直接写入 `contacts.json` 联系人对象 | 单文件易查询，避免引入新数据文件；与 tags/notes 同级 |
| `goals.yaml` 路径 | `config/goals.yaml`（支持 `goals.local.yaml` 覆盖） | 与 `config.yaml` 同目录，复用配置加载模式 |
| AI 建议生成 | `ai.suggest_leverage()` 复用 `_call_llm`，失败降级为基于 tags/relation 的规则推断 | 兑现"两条路径降级"架构承诺；规则降级保证离线可用 |
| 批量默认值 | `--batch 5`（非 SPEC 草案的 10） | 单次 CLI 调用 5 人避免疲劳；`--all` 可一次过完全部候选 |
| `--confirm` 标志 | 跳过交互直接写入 AI 建议（默认不开） | 提供批量自动化通道，但默认走人工确认（原则二） |
| `confirmed` 语义 | `confirmed` 为空 = 仅 AI 建议、未生效；`list_unanchored` 把这类视为待锚定 | 严格区分"建议"与"已确认"，未确认可重新建议 |
| 储备池强制锚定 | `--force` 才能锚定 reserve 联系人 | SPEC §18.3 仅 core 层锚定为默认，留逃生口但不鼓励 |

## §19. v4.0 支柱二：建议引擎（承诺②升级）

### 19.1 定义

把"该联系谁"升级为"**联系谁 + 为什么 + 聊什么**"三元组：

```
建议 = f(冷却状态, health分, important_dates, leverage锚定, timeline上下文, todos)
```

所有信号源均为已有数据，无新采集需求（原则六：信号先行）。

### 19.2 输出形态

新命令 `social advise`，输出本周 3-5 条经营动作，每条含三元组：

```
📋 本周经营建议（2026-07-13）
1. 联系姜少 —— 21天未联系🔴 + 锚定[事业]；
   上次聊到XX项目进展，可跟进结果 → social draft -c 姜少
2. 联系王总 —— 下周三生日 + 强度4；备好祝福
3. ...
```

**约束**：
- 每周建议条数 3-5 条封顶——经营动作以周为单位才可持续，不制造焦虑
- 每条建议必须带"为什么"（可追溯到具体信号）和"聊什么"（可一键接 draft）
- 建议仅建议，不自动创建待办（核心规则3的精神延伸）

### 19.3 推送节奏（✅ 已确认 2026-07-10）

- **周推**：每周一晨间推送"本周经营简报"（`social advise` 输出）
- 每日晨推（7:33）维持现状（当日日程+待办），不并入周报内容
- 这是初心里"自动"二字的真正含义：**自动化的是思考，不只是提醒**

## §20. v4.0 支柱三：兑现追踪（原则四落地）

### 20.1 定义

timeline 的 `type` 枚举新增 `outcome`（成果）：某段关系产生引荐/信息/合作机会/实际帮助时记一笔。

```json
{
  "id": "t-xxxxx",
  "date": "2026-08-01",
  "contact": "xxx",
  "type": "outcome",
  "summary": "经张总引荐认识XX行科技部负责人",
  "goal": "事业"
}
```

### 20.2 规则

- `goal` 字段关联到六维目标之一（可空）
- **只做可追溯记录，不做 ROI 计算**（原则七：保守推理，可追溯）
- 价值：半年后可回答"我在人脉上投入的时间换来了什么"
- 查询：`social outcomes [--goal 事业] [--year 2026]`

## §21. 范围边界与版本管理

### 21.1 明确不做（防止范围回潮）

| 功能 | 状态 | 理由 |
|:----|:---:|:----|
| `_connections` 关系图谱 | ❄️ 冻结至 v4.x+ | 零数据基础；服务"存得多"，v4.0 主线是"用得深" |
| `_groups` 群感知 | ❄️ 冻结 | 同上 + 微信API接入复杂 |
| `import-chat` / `import-calls` | ❄️ 冻结 | 合规风险 + "记一下"摩擦已低 |
| 批量 enrich 灰色联系人 | ⬇️ 降级 | 已被实测证伪，改为晋升触发式单人补全（§17） |

### 21.2 版本号论证

定 **4.0** 而非 3.5：按 SPEC §5.1，数据模型重大变更（leverage 字段 + timeline 新类型 + 建议引擎）触发主版本；"从温度计到参谋"值得整数里程碑。
