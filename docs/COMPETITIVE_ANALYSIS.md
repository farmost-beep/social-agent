# social-agent 竞品深度研究

> **本文档是产品决策依据，不是营销材料。**
> 调研覆盖 15+ 竞品，数据来源为各产品官网、App Store、第三方评测（2025-2026）及用户反馈。
> 最后更新：2026-07-10

---

## 一、市场全景：四个赛道，同一痛点

"社交关系管理"产品可分为四个赛道，看似同质但定位根本不同。

### 赛道 A：社交平台（微信 / LinkedIn / 朋友圈）

| 维度 | 微信 | LinkedIn |
|:----|:----|:----|
| 本质 | **通信管道** + 社交广场 | **职业档案柜** + 弱关系广场 |
| 关系管理 | 无——通讯录是平的，没有分层/强度/目标 | 无——connections 是计数器，不是关系深度 |
| 核心动作 | 聊天、朋友圈、群 | 发帖、连接、求职 |
| 痛点 | 4683 个联系人平铺，找不到"该联系谁"；朋友圈是广播不是经营；没有"上次聊了什么"的记忆 | 连接数是虚荣指标；500+ connections 但 90% 从不互动；没有"该经营谁"的判断 |

**关键诊断**：微信和 LinkedIn 是**社交基础设施**，不是关系管理工具。它们解决了"连接"问题，但完全不解决"经营"问题。用户的 4683 个微信联系人就是证据——存得下，用不起来。

### 赛道 B：个人 CRM（Clay / Dex / Monica / Norrie / Keep / 脉客）

2021 年后兴起的新赛道，已高度拥挤：

| 产品 | 定位 | 核心机制 | 价格 | 数据来源 |
|:----|:----|:----|:----|:----|
| **Clay** (→Mesh) | "自动更新的关系网" | 邮件/日历/LinkedIn/Twitter 自动导入+enrichment | $120/年 | 自动采集 |
| **Dex** | "有节奏地保持联系" | LinkedIn+Gmail 同步+follow-up cadence | $144/年 | 半自动 |
| **Monica** | "开源私有日记式 CRM" | 手动记录活动/礼物/债务/家庭 | $90/年或自托管 | 全手动 |
| **Norrie** | "真实生活的人际管理" | packs of memory（宠物/礼物/人生事件） | 20人免费 | 全手动 |
| **Keep** | "语音优先的 AI CRM" | 说10秒→AI 提取人/日期/跟进 | iOS | 语音 |
| **脉客 Connectors** | "AI 人脉教练" | 名片扫描+互动记录+案件漏斗 | 800人免费 | 半自动 |
| **钛合金人脉** | "5+50+100 人脉圈" | 基于《给予者》方法论 | — | 全手动 |
| **Yenesow** | "AI-first 个人 CRM" | AI 健康分+对话开场白+关系图谱 | iOS | 全手动+AI |
| **Orbit** | "AI 捐赠者关系管理" | 面向非营利组织捐赠者经营 | 企业级 | 半自动 |
| **Dayze** | "时间线个人 CRM" | 关系绑定人生天数+里程碑 | — | 手动 |

**赛道共性**：全部聚焦"记住上次聊了什么 + 提醒该联系谁"——即 social-agent v2.x 的水平。

**赛道分化轴**（OpenLoop 2026 总结）：

1. **自动同步 vs 手动记录**：Clay 自动 vs Monica 全手动
2. **职业 vs 全生活**：Dex 职业偏向 vs Norrie 生活偏向
3. **AI 深度**：Keep/Yenesow AI-first vs Monica 几乎无 AI

### 赛道 C：企业 CRM 降维（Salesforce / HubSpot / Zoho 个人版）

| 维度 | 特征 |
|:----|:----|
| 本质 | 销售漏斗管理 |
| 关系定义 | 交易关系（lead → opportunity → deal） |
| 痛点 | 字段太多太乱、录入体验极差、被当监控而非工具 |

**Zoho 调研数据**：64% 单用户 CRM 用户"功能不够用"，但更常见是"用着用着就弃了"。核心原因：录入摩擦 > 感知价值。

**关键诊断**：企业 CRM 的"漏斗"心智模型不适用于个人关系——人际关系不是 lead → deal 的线性管道。

### 赛道 D：AI 对话辅助（Convo / Qubi / Diggsi）

| 产品 | 定位 |
|:----|:----|
| Convo | 会议中实时建议 + 会后自动跟进 |
| Qubi | 销售通话实时教练 |
| Diggsi | LinkedIn 资料分析 → 对话开场白 |

**关键诊断**：这些是**单次对话的辅助**，不是长期关系管理。它们解决"这次聊什么"，但不解决"这半年该经营谁"。

## 二、social-agent 的定位差异：五个独有维度

把 social-agent 放在上述四个赛道中，差异不是功能层面的，而是**哲学层面的**。

### 差异 1：目标锚定——从"记住谁"到"为什么经营"

| 维度 | 所有竞品 | social-agent v4.0 |
|:----|:----|:----|
| 关系管理的基础 | "上次聊了什么" + "多久没联系" | **"这个人对我哪个目标有贡献" + "怎么经营"** |
| 数据模型 | timeline + reminder | **leverage{goals, how, direction, confirmed}** + timeline |
| 建议引擎输入 | 冷却状态、生日 | 冷却 + 生日 + **leverage 锚定** + 待办 + 强度（五信号源） |
| 建议输出 | "3个月没联系了，该问候一下" | **"联系谁 + 为什么（锚定目标）+ 聊什么（具体话题）"三元组** |

**这是所有竞品都没有的维度。** Clay 的 enrichment 是"对方换了工作"——关于对方的事实；social-agent 的 leverage 是"这个人能帮我撬动事业目标"——关于**关系与我的目标的映射**。前者是信息，后者是策略。

调研中最接近的是"钛合金人脉"（基于《给予者》5+50+100 方法论），但它只做了分层，没做目标映射。askcruit 的"Track Goals, Not Dates"理念相近，但那是博客建议，不是产品功能。

### 差异 2：社交价值理论——从"情感亲疏"到"目标贡献度"

| 维度 | 传统社交直觉 | 所有竞品 | social-agent |
|:----|:----|:----|:----|
| 强度定义 | 关系好坏/感情深浅 | 互动频率（Clay）/ 手动标签（Dex）/ 无（Monica） | **社交价值 = 对用户六维目标的贡献度** |
| 强度排序 | 按感情 | 按互动频率 | **按目标撬动力** |
| 强度调整 | 不调整 | Clay 按互动频率自动调整 | **按社交价值变化调整（换工作/合作深化）** |

**这是理论层面的根本差异。** 竞品的"关系强度"是互动频率的副产品，social-agent 的"强度"是社交价值的显式评估。前者回答"我们互动多吗"，后者回答"这个人对我重要吗"。

### 差异 3：建议引擎——从"提醒"到"参谋"

| 维度 | Dex/Clay/脉客 | Yenesow/Keep | social-agent v4.0 |
|:----|:----|:----|:----|
| 建议类型 | "该联系了"（提醒） | "可以聊这个"（开场白） | **"联系谁 + 为什么 + 聊什么"三元组** |
| 建议依据 | 时间间隔 | AI 生成 | **五信号源打分 + leverage 锚定** |
| 建议深度 | "3个月没联系" | "他儿子要高考了" | **"锚定[事业/AI能力] → 以AI审计方案请教"** |
| 建议数量 | 全量提醒 | 不限 | **每周 3-5 条封顶（聚焦）** |
| 建议后衔接 | 无 | 无 | **附 `social draft` 命令，一键拟稿** |

**关键差异**：竞品的"建议"本质是**提醒**（reminder），social-agent 的"建议"本质是**策略建议**（advise）。提醒说"该做了"，策略说"该做什么 + 为什么 + 怎么做"。

### 差异 4：兑现追踪——从"记录"到"闭环"

| 维度 | 所有竞品 | social-agent v4.0 |
|:----|:----|:----|
| 记录什么 | 互动记录（聊了什么） | 互动记录 + **兑现成果（outcome）** |
| 回顾维度 | "上次联系是什么时候" | **"这半年在人脉上投入的时间换来了什么"** |
| ROI | 无 | **明确不做 ROI 计算（原则七），只做可追溯记录** |

**这是所有竞品完全没有的维度。** 没有产品回答"我经营人脉的回报是什么"。Monica 记录"活动"，Dex 记录"互动"，但没人记录"这次互动产出了什么成果"。

### 差异 5：本地优先 + CLI 原生——从"云端 SaaS"到"数据主权"

| 维度 | Clay/Dex/脉客 | Monica | social-agent |
|:----|:----|:----|:----|
| 数据存储 | 云端（加密） | 云端或自托管 | **本地 JSON，不上云** |
| 数据主权 | 平台持有 | 用户持有（可自托管） | **用户完全持有** |
| 界面 | GUI（App/Web） | Web | **CLI 优先** |
| 集成 | 邮件/日历/LinkedIn API | 无 | **微信推送（可选）+ LLM 抽象层** |
| 定价 | $120-144/年 | $90/年或免费 | **开源免费** |

**关键差异**：social-agent 是**中国语境下的本地优先工具**。Clay/Dex 依赖 Gmail + LinkedIn 生态，在中国不可用（LinkedIn 已退出中国，Gmail 不普及）。social-agent 的微信推送 + 本地存储 + CLI 是唯一适配中国社交生态（微信主导）的方案。

## 三、定位矩阵

```
                    关系深度管理
                         ↑
                         |
          Monica/Norrie  |  social-agent v4.0
          (手动记录生活)  |  (目标锚定+建议引擎+兑现追踪)
                         |
  互动频率 ──────────────┼────────────── 目标对齐
                         |
          Dex/脉客       |  (空缺)
          (节奏提醒)      |
                         |
          Clay           |  (空缺)
          (自动enrichment)|
                         ↓
                    信息聚合管理
```

**social-agent 独占"目标对齐 × 关系深度管理"象限。** 没有任何竞品同时做到：

1. 以用户目标为锚点定义关系价值
2. 以策略建议（非提醒）驱动经营行动
3. 以兑现追踪闭合"想 → 做 → 得"循环

## 四、痛点诊断：三层痛点

### 第一层：现有产品共同痛点（赛道级）

| 痛点 | 证据 | 谁解决了 |
|:----|:----|:----|
| **数据录入摩擦** | 范凯："Excel 填完半年就不更新了"；Zoho：64% 用户"用着用着就弃了" | Clay 自动同步（但依赖 Gmail/LinkedIn，中国不可用）；Keep 语音（但仅 iOS） |
| **"该联系谁"无策略** | 全部产品只做"时间到了"提醒，不做"为什么该联系" | **无人解决** |
| **关系强度无标准** | Clay 按频率自动算，Dex 手动标，Monica 无 | **无人解决**——social-agent 的社交价值理论是唯一显式框架 |
| **无目标对齐** | 没有产品问"你的人脉服务于什么目标" | **无人解决**——social-agent v4.0 leverage 是独创 |
| **无兑现闭环** | 没有产品追踪"人脉投入产出什么" | **无人解决**——social-agent v4.0 outcome 是独创 |
| **中国生态缺失** | Clay/Dex 依赖 Gmail + LinkedIn，中国用户用微信 | **无人解决**——social-agent 微信推送是唯一适配 |

### 第二层：social-agent 独有痛点（自身诊断）

| 痛点 | 数据证据 | 根因 |
|:----|:----|:----|
| **timeline 真空** | 核心圈 317 人中 99% 从未在 timeline 出现，32 条/4个月 | 手动记录摩擦仍太高（比竞品好但不够好）；import-chat/calls 被冻结 |
| **锚定 ≠ 经营** | 31 人已锚定但 30 人零 timeline 互动记录 | 锚定是"想清楚"，但没解决"动起来"；advise 对零 timeline 的人只能空猜"聊什么" |
| **outcome 空转** | 0 条 outcome 记录 | 兑现追踪是 v4.0 最超前的设计，但用户还没有"记录成果"的习惯 |
| **CLI 摩擦** | 每次记录要打开终端输入命令 | 相比 Clay 的"邮件自动同步"或 Keep 的"说10秒"，CLI 是逆向选择 |
| **4366 reserve 联系人沉睡** | reserve 层 4366 人，enrich 仅 3 人 | 批量 enrich 路线被证伪（一个月仅 3/4683），晋升触发式尚未跑通 |

### 第三层：用户（陈颖芳）特有痛点

| 痛点 | 证据 |
|:----|:----|
| **邮储体系内关系经营无工具** | 8 个强度5 的上级领导 + 10+ 总行/分行高管，没有工具帮"想清楚怎么经营上级" |
| **跨界身份整合难** | 邮储 + 科大校友 + 民建 + 创业四个圈层交叉，竞品无法处理多维角色 |
| **投资 + 事业双重目标** | leverage 中投资维度仅 2 人锚定，但实际有昆仑万维重仓等投资决策需要人脉支撑 |

## 六、结论：差异本质

**social-agent 不是"更好的个人 CRM"，而是一个不同物种。**

| 心智模型 | 个人 CRM | social-agent |
|:----|:----|:----|
| 隐喻 | 通讯录++（记住更多人） | **参谋（帮你想清楚经营谁、为什么、怎么经营）** |
| 核心问题 | "上次聊了什么" | **"这个人对我哪个目标有贡献，怎么经营"** |
| 价值主张 | "不忘掉重要的人" | **"把人脉投入对准人生目标，并追踪兑现"** |
| 终极指标 | 互动频率 / 联系覆盖率 | **目标维度的人脉兑现率** |

### 三个竞品完全没有、social-agent 独有的功能

1. **leverage 锚定**——关系 → 目标映射（SPEC v4 §18）
2. **advise 三元组**——联系谁 + 为什么 + 聊什么（SPEC v4 §19）
3. **outcome 追踪**——人脉投入的成果闭环（SPEC v4 §20）

### 三个所有产品都有、social-agent 反而最弱的

1. **数据采集**——Clay 自动同步 vs social-agent 全手动（timeline 真空是证据）
2. **GUI 体验**——Clay/Dex 精美 App vs social-agent CLI
3. **移动端**——所有竞品有 App vs social-agent 无

### 一句话总结

social-agent 在"想清楚"层面领先所有竞品 2-3 个代际（目标锚定 + 策略建议 + 兑现追踪），但在"动起来"层面落后 Clay 一个代际（数据采集摩擦）。这正是 v5.0 要补的课——不是追赶 Clay 的自动同步，而是用"轻量记录 + 信号推断"走第三条路。

## 七、对 v5.0 演进的产品决策启示

基于竞品调研，v5.0 的三个支柱有明确的市场对标：

| v5.0 支柱 | 对标竞品 | 差异化策略 |
|:----|:----|:----|
| **轻量记录（touch 类型 + log 简化）** | Keep 的语音 / Clay 的自动同步 | 不走自动同步（中国不可用），不走语音（CLI 无麦克风），走"极低摩擦键盘输入" |
| **信号推断（infer 引擎）** | Clay 的自动 enrichment | Clay 推断的是"对方信息"（职位/公司），social-agent 推断的是"互动模式"（可能的经营节奏） |
| **执行闭环（review + advice-id）** | 无竞品 | 独创——所有竞品都没有"建议 → 执行 → 反馈"闭环 |

**v5.0 不做的事（基于竞品调研确认）**：

| 不做 | 理由 |
|:----|:----|
| GUI/App | Clay/Dex/脉客已充分覆盖 GUI 市场；social-agent 的 CLI 是数据主权定位的体现，不是缺陷 |
| Gmail/LinkedIn 自动同步 | 中国生态不可用；且自动同步与"显式确认"原则（原则二）冲突 |
| 关系图谱可视化 | Dex 的关系图谱被评测为"barely readable at scale"；数据真空下无意义 |
| 语音输入 | Keep 已做且仅 iOS；CLI 场景下键盘输入更快 |
