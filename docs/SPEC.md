# 社交关系AI管家 产品设计规约 (SPEC)

> 本规约定义产品的设计理念、数据模型、操作规则和演进原则。
> 所有对该系统的修改、查询、扩展均须遵循本文档。

---

## 0. 理论基础

### 0.1 关系强度的定义：社交价值

社交关系强度按**社交价值**排序——即这个人对用户**实现个人目标**的贡献度。

用户可自定义自己的多维目标框架。默认示例如下：

1. **事业发展** — 职业晋升、业务拓展、能力提升
2. **财富增长** — 投资回报、收入增长、财务安全
3. **家庭和谐** — 子女成长、夫妻关系、父母赡养
4. **身心健康** — 运动习惯、体检指标、心理健康
5. **知识精进** — 阅读体系、技能习得、认知升级
6. **社交网络** — 人脉拓展、关系维护、资源整合

社交价值的大小，取决于这个人能帮助撬动目标中的哪一个、撬动多大。

| 强度 | 社交价值等级 | 典型人物 |
|:----:|:-----------|:--------|
| **5** | **能直接影响决策 + 家人** | 直接上级、关键决策者、配偶、子女 |
| **4** | **能深度合作/提供重要机会** | 资深同行、创业伙伴、重要前领导 |
| **3** | **有价值的信息/资源交换对象** | 同业校友、行业专家、合作方 |
| **2** | **普通社交关系** | 一般校友、泛泛之交 |
| **1** | **弱关系** | 仅存联系方式，几乎无互动 |

### 0.2 社会角色框架

社交关系的本质是**社会角色与角色之间的互动**。每个联系人不是单一的人，而是以某个角色身份与你互动。

用户自定义社会角色体系：

| 层级 | 角色类别 | 由用户自定义 |
|:----:|:--------|:------------|
| **一级** | 🏢 当前职业身份 | 当前所在单位/职位 |
| | 🏛️ 前职业身份 | 过往重要职业经历 |
| **二级** | 🎓 教育背景 | 毕业院校/专业 |
| | 🤝 社会身份 | 党派/协会/社团 |
| | ✍️ 专业领域 | 研究方向/专业特长 |
| **三级** | 👨‍👩‍👧‍👦 家庭角色 | 家庭中的身份 |
| | 💰 理财身份 | 投资偏好/财务目标 |
| | 🎯 个人标签 | 兴趣爱好/个人标签 |

**社交价值公式：**

```
社交价值 = Σ (我的角色A × 对方角色B × 双向互动力)
```

- 角色的匹配度决定了**互动的可能性**（同行高管×同行高管 = 同体系，互动成本低）
- 双向互动力决定了**互动的可实现性**（某行业大咖潜在价值5，但双向互动力不足，实际只到4）
- 强度应反映**在特定角色关系下已实现的互动价值**

### 0.3 角色互动层数

社交关系质量由**角色互动层数**决定——即双方在多少个不同社会角色维度上存在交集。

**角色维度定义：**

| 维度 | 标签 | 含义 |
|:----:|:----:|:----|
| **职业层** | `职层` | 当前/过去在同一单位或行业 |
| **校友层** | `校层` | 毕业院校校友 |
| **组织层** | `组层` | 同社团/协会成员 |
| **社交层** | `群层` | 同在某个微信/社交群 |
| **合作层** | `合层` | 有项目合作关系 |
| **家庭层** | `家层` | 配偶、子女、父母等家人 |

**层数越高，社交交集越密：**

| 层数 | 含义 | 系统标记 | 示例 |
|:---:|:----|:--------|:----|
| **3层** | 三重角色叠加 | `角色x3` | 用户A（校友+同行+群友） |
| **2层** | 双重角色叠加 | `角色x2` | 用户B（现领导+校友） |
| **1层** | 单层角色 | `角色x1` | 家人、纯校友、纯群友 |

角色互动层数作为辅助参考，不直接决定强度，但与社交价值正相关。

### 0.4 核心原则

**原则一：社交价值排序。** 强度高 = 这个人在你的事业/业务/信息网络中位置更重要。不按情感亲疏排。

**原则二：强度不可批量设定。** 每段关系的强度必须由用户逐一确认，不能根据公司名、学校名等标签批量赋值。

**原则三：强度随社交价值变化而变。** 这个人换工作、升职、或合作深化了，强度就应该调整。

**原则四：潜在价值≠实际价值。** 社交价值需要**双向互动**才能兑现。一个人理论上能帮你（潜在价值5），但如果没有实际互动和关系深度（双向互动），实际社交价值不应高于4。

**原则五：关系需要持续经营。** 长期不互动会冷却。冷却检查（14天预警、21天告警）体现的就是"价值关系需要维护"。

## 1. 核心设计理念

### 1.1 以人+关系为核心

系统不是通讯录，而是**关系网**。每个联系人的核心价值不是他的职务有多高，而是**他跟用户的连接是什么**。

- 记录一个人时，优先回答：**"我怎么认识他的？"** 而非 "他是干什么的？"
- 关系强度（strength）必须反映**用户与该人的实际关系深浅**，而非根据单位/职务批量设定
- 强度按社交价值排序，不按情感亲疏

### 1.2 统一数据源

**所有社交关系只记录在此系统内**，不分散存储在不同地方。

| ✅ 正确做法 | ❌ 错误做法 |
|-----------|-----------|
| 社交关系 → data/contacts.json | 社交关系 → memory/*.md |
| 用 `note` 命令加结构化记忆 | 用通用记忆文件存储社交信息 |

### 1.3 社交资源的定义

"社交相关资源"、"社交关系"、"认识的人"、"校友资源"、"人脉"等，指的都是**记入此系统的关系**，而非泛指所有人。只有经过筛选、评估、结构化记录的社交关系才算"社交资源"。

## 2. 数据模型

### 2.1 联系人 (Contact)

```json
{
  "id": "唯一标识符",
  "name": "显示名称",
  "relation": "关系分类 (self/family/校友/同行/创业/合作/其他)",
  "sub_relation": "子分类",
  "strength": 1-5,
  "tags": ["标签1", "标签2"],
  "platforms": { "weixin": "", "phone": "", "email": "" },
  "notes": "自由文本备注",
  "memories": [],
  "important_dates": [],
  "stage": "",
  "created": "创建日期"
}
```

### 2.2 强度 (strength) 定义

| 强度 | 社交价值 | 示例 |
|:----:|:--------|------|
| **5** | **直接影响决策** | 直接上级、关键决策者、重要客户 |
| **4** | **重要合作/引路人** | 前领导、创业伙伴、资深同行 |
| **3** | **有价值的信息交换** | 同业校友、行业专家、潜在合作方 |
| **2** | **普通社交关系** | 一般校友、泛泛之交 |
| **1** | **弱关系** | 仅存联系方式，几乎无互动 |

**重要规则：** 强度不应批量设定。每个人的强度应基于其对用户的**社交价值**逐一评估。

### 2.3 记忆 (Memory)

```json
{
  "id": "mem-xxxxx",
  "content": "记忆内容",
  "tags": ["标签"],
  "created": "创建时间"
}
```

使用场景：记录履历、共同经历、重要事件、个人特征等。

### 2.4 时间线 (Timeline)

```json
{
  "id": "t-xxxxx",
  "date": "日期",
  "contact": "联系人ID",
  "type": "类型 (message/meeting/call/milestone)",
  "summary": "摘要",
  "pending": "待办事项",
  "key_points": ["关键点"]
}
```

## 3. 操作规则

### 3.1 查询规则

涉及联系人数据的查询，**禁止从记忆或上下文猜答案**，必须读 `contacts.json` 获取真实数据。

### 3.2 修改规则

- 新增联系人 → `add-contact` 命令
- 更新联系人 → `edit-contact` 命令
- 添加记忆 → `note` 命令
- 记录互动 → `log` 命令

### 3.3 关系维护规则

- 每次添加记忆（note）时，如果关系强度偏低（<5），自动+1
- 超过**14天**未联系的关系标记为🟡冷却
- 超过**21天**未联系的关系标记为🔴预警
- 生日可从 notes 中自动提取（如"1129"→11月29日）

## 6. 演进原则

1. **增量演进**：不追求一次性完美，每次交互中持续优化
2. **数据驱动**：所有设计决策基于实际使用反馈，而非预设
3. **人本优先**：任何技术优化不能削弱"以人+关系为核心"的设计理念
4. **最小冗余**：一条信息只在一个地方存储，不跨系统重复
5. **查询优先**：优化读端体验，写端可接受适当延迟

## 7. 自动化规则

### 7.1 基于时间线的强度动态调整

**降级触发（长期无互动）：**

| 当前强度 | 90天无互动 | 180天无互动 |
|:--------:|:----------:|:-----------:|
| **4** | 建议降为3 | 建议降为2 |
| **3** | 建议降为2 | 建议降为1 |

**升级触发（高频互动）：**
- 近**60天内≥2次**互动，且当前强度<4 → 建议升1级（不超过4）

**保护规则：**
- 强度5（决策层+家人）：**永不自动调整**
- 强度2及以下：**永不自动调整**
- self（自己）：**永不自动调整**

**执行方式：**
- `adjust` → 查看建议
- `adjust --apply` → 确认执行

### 7.2 记忆备注的轻度提升

添加记忆（`note`）时自动提升强度1级，但不超过4，5级家人除外。

## 8. v2.5.0 新增功能

> 版本定位：增量升级，聚焦"从联系人数据库到社交活跃伙伴"的转化。

**核心问题：** v2.4 积累了 3765 位联系人，但其中 3012 人（80%）处于强度 2 的"灰色地带"——仅存姓名和最低限度的导入信息，缺乏画像、互动记录和关系判定。系统"存得下来"，但"用不起来"。

**v2.5 目标：** 将灰色联系人变为可经营的关系资产，同时建立关系之间的连接感知。

| 优先级 | 功能 | 解决什么 |
|:-----:|:----|:--------|
| **P0** | 批量画像补全管道 | 3012 个"灰色联系人"自动补全行业/角色/标签 |
| **P1** | 关系图谱 | 谁认识谁——打通联系人之间的连接孤岛 |
| **P1** | 群聊智能感知 | 联系人来自哪个群、谁和谁同群 |
| **P2** | 互动信号自动采集 | 微信聊天/通话记录自动导入，降低记录摩擦 |
| **P2** | 关系健康分 | 综合评分替代二元冷却检查，主动建议维护 |

### 8.2 功能一：批量画像补全管道 (P0)

#### 8.2.1 目标

对系统中缺乏结构化信息的联系人（以强度≤2、无角色分类、无标签的"灰色联系人"为主），利用 AI 推理自动补全：**角色分类、子分类、行业标签、简要画像备注**。

#### 8.2.2 输入信号

管道对每个候选人收集以下上下文：

| 信号 | 来源 | 判别力 |
|:----|:----|:-----:|
| 联系人姓名 | contacts.name | 中——姓名可暗示性别/地域/辈分 |
| 已有备注 | contacts.notes | 高——OCR导入常有碎片信息 |
| 已有标签 | contacts.tags | 高——已有标签是最可靠的信号 |
| 已有记忆 | contacts.memories[].content | 高——历史记录中的人物描述 |
| 所属群组 | contacts._groups（若已填充） | 高——群名可推断行业/组织 |
| 微信 ID 特征 | contacts.id 或 wechat_ids | 低——wxid 格式无信息量 |
| 姓名 + ID 联合模式 | 特殊前缀/后缀 | 低——OCR导入的联系人 ID 有时含组织缩写 |
| 🌐 **网络公开信息** | DuckDuckGo 搜索引擎（`--web` 模式） | 高——可获取公司/职位/行业等公开资料，显著提升置信度 |

#### 8.2.3 推理目标

对每个联系人，AI 输出以下字段：

| 输出 | 说明 | 写入位置 |
|:----|:----|:--------|
| `relation` | 角色分类：同行/校友/合作/家人/其他 | contacts.relation |
| `sub_relation` | 子分类（如：银行/量化/创业/学术） | contacts.sub_relation |
| `suggested_tags` | 推理得到的标签列表 | contacts.tags（追加） |
| `notes_append` | 推理总结备注 | contacts.notes（追加，前缀"[AI补全]"） |
| `confidence` | 整体置信度 1-10 | 仅在日志中记录 |

**置信度规则：**

| 置信度 | 含义 | 行为 |
|:-----:|:----|:----|
| **8-10** | 强证据（群名含行业 + 备注含职位） | 写入 relation 和 tags |
| **5-7** | 中等证据（姓名 + 群名可推断） | 仅追加 notes（"[AI补全] 推测..."），不覆盖 relation |
| **1-4** | 弱证据（仅有姓名） | 跳过，不写入任何内容 |

**保护规则：**
- **relation 覆盖保护：** 已有非空 relation 的联系人，AI 不覆盖，仅在 notes 追加
- **notes 不删除已有内容：** 永远 append，不 replace
- **strength 不受补全影响：** 补全不改变强度值
- **tags 只追加不覆写：** 新标签与旧标签并集

#### 8.2.4 CLI 接口

```bash
# 单次运行：处理 10 个最高优先级的未处理联系人
python3 src/social.py enrich

# 指定批次大小
python3 src/social.py enrich --batch 20

# 仅检查不动手（预览模式）
python3 src/social.py enrich --dry-run

# 重新处理已处理过的联系人
python3 src/social.py enrich --force

# 只看统计
python3 src/social.py enrich --stats

# 启用外部网络信源（通过 DuckDuckGo 搜索补充上下文）
python3 src/social.py enrich --web

# 组合使用
python3 src/social.py enrich --web --batch 20 --force
```

#### 8.2.5 优先级排序

管道处理顺序按**信息密度从高到低**排列：

1. 强度 ≥ 3 但 relation 为空（高价值但未分类）
2. 有 notes 但 relation 为空
3. 有 tags 但 relation 为空
4. 有 memories 但 relation 为空
5. 仅存姓名 + wxid（最低优先级，可能跳过）

#### 8.2.6 外部网络信源（`--web` 模式）

通过 DuckDuckGo HTML 搜索（免费，无需 API Key）获取联系人公开信息，注入 AI 提示词作为额外上下文。

**搜索策略：**
1. 从本地数据中提取搜索上下文关键词（标签、备注、角色）
2. 使用 `{姓名} {上下文}` 构造搜索查询
3. 获取前 3 条搜索结果（标题 + 摘要片段）
4. 结果注入 AI 提示词 `网络公开信息（搜索结果）：` 章节
5. 同一批次内相同查询自动缓存，避免重复搜索

**效果说明：**
- 对已有一定本地上下文的联系人（有备注/标签/角色），网络搜索能显著提高推理置信度
- 对仅有姓名的弱关系联系人，搜索可能返回噪声，AI 会自动降权处理
- 速率限制：300ms 间隔，避免触发搜索引擎反爬

#### 8.2.7 安全与幂等

- 每次运行写入 `data/enrichment_log.json`，可追溯
- 已处理联系人的 `_enrich_version` 递增
- `--force` 重新处理时清空该联系人的旧日志并重新推理
- 管道可随时中断，下次从断点继续（按优先级排序，已处理的不重复）

#### 8.2.7 统计视图

```bash
python3 src/social.py enrich --stats
# 输出示例：
# 已补全: 45人
# 待补全: 2967人
# 按置信度: 高 12人 / 中 33人 / 低 0人
# 按角色: 同行 20人 / 校友 15人 / 其他 10人
```

### 8.3 功能二：关系图谱 (P1)

#### 8.3.1 目标

在联系人之间建立**显式连接**。支持查询"谁认识谁"、"如何通过关系网接触到某人"。

#### 8.3.2 数据模型

见 8.1.2 Connection 结构。每个联系人的 `_connections` 数组记录其直接关系。

**双向一致性规则：**
- 添加 A→B 的连接时，系统**自动同步**添加 B→A 的连接（target_id 互换，type 不变）
- 删除连接同理双向删除
- 避免手动维护不同步

```json
// 添加连接前
A._connections = []
B._connections = []

// 执行 add-connection A B --type 同事
A._connections = [{target_id: "B", type: "同事", ...}]
B._connections = [{target_id: "A", type: "同事", ...}]  // 自动同步
```

#### 8.3.3 CLI 接口

```bash
# 添加连接
python3 src/social.py connect <contact> <target> --type TYPE [--notes NOTES]

# 列出联系人的关系网
python3 src/social.py connections <contact>

# 查找两人之间的路径
python3 src/social.py find-path <contact> <target>

# 路径查找示例：
# $ social.py find-path 张三 李四
# 张三 → 王五（同事）→ 李四（校友）
# 共 2 跳
```

#### 8.3.4 图表可视化

使用现有 matplotlib + networkx 基础设施生成关系图谱：

```bash
python3 scripts/graph.py                     # 生成完整图
python3 scripts/graph.py --contact 张三       # 仅显示张三的 ego 网络（2 跳内）
```

生成的 PNG 保存到 `data/` 目录，与现有 `relationship_graph.png` 一致。

#### 8.3.5 仪表盘集成

仪表盘新增：
- 总连接数
- 连接密度（connections / contacts 比率）
- 最大连接数的联系人 TOP5

### 8.5 功能四：互动信号自动采集 (P2)

#### 8.5.1 目标

降低记录互动的摩擦。用户不需要手动输入"记一下：和××聊了什么"，而是通过导入既有数据源自动生成时间线。

#### 8.5.2 微信聊天记录导入

```bash
python3 src/social.py import-chat <file.txt>
```

**输入格式支持：**
- 微信 PC 版导出的 `.txt` 格式
- 格式示例：`2026-06-14 14:30 张三 说：项目方案收到了`
- 兼容带时间戳的常见聊天导出格式

**处理逻辑：**
1. 扫描文件中的每一条消息
2. 提取发送者姓名 → 匹配 `contacts.json` 中的联系人
3. 按日期 + 联系人聚合为 timeline 记录
4. 多条消息聚合为一条 `summary`（如"聊了项目方案"）
5. 自动去重：同一日期 + 同一联系人 + 内容相似度 > 80% 视为重复

**限制与保护：**
- 仅处理与已知联系人匹配的消息，不自动创建新联系人
- 每次导入生成 `data/import_chat_log.json` 记录

#### 8.5.3 通话记录导入

```bash
python3 src/social.py import-calls <file.csv>
```

**输入格式支持：**
- iOS 通讯录导出 CSV（姓名, 日期, 时长, 方向）
- Android 通话记录 CSV
- 通用格式：姓名, 日期, 方向（呼出/呼入）

**处理逻辑：**
1. 匹配联系人姓名
2. 按日期 + 姓名聚合为 timeline 记录（type=call）
3. 去重：同一人同日多次通话聚合为"通话 X 次"
4. 不记录通话内容，仅记录时间与频率

#### 8.5.4 安全规则

- 导入仅 append 时间线，不修改联系人数据
- 导入日志记录每次操作的行数、匹配数、新增数
- 支持 `--dry-run` 预览

### 8.6 功能五：关系健康分 (P2)

#### 8.6.1 目标

替代当前二元的"14天/21天冷却检查"，提供一个**综合关系健康评分**（0-100），量化每段关系的维护状态，并给出可操作的建议。

#### 8.6.2 评分模型

```
health_score = recency × 0.40 + depth × 0.30 + layers × 0.20 + events × 0.10
```

| 因子 | 权重 | 计算方式 | 得分区间 |
|:----|:----:|:--------|:--------:|
| **recency** | 40% | 最后互动距今天数映射：≤7天=100，≤14天=80，≤30天=60，≤90天=40，>90天=20 | 20-100 |
| **depth** | 30% | 最近 3 次互动的类型深度：见面=100，通话=70，消息=40，无记录=0 | 0-100 |
| **layers** | 20% | 角色互动层数 × 33：x3=100，x2=66，x1=33，无=0 | 0-100 |
| **events** | 10% | 近期有生命周期事件（生日/换工作等）=100，无=50 | 0-100 |

**总分对照：**

| 健康分 | 等级 | 含义 | 建议动作 |
|:-----:|:----:|:----|:--------|
| **80-100** | 🟢 健康 | 最近有互动，关系活跃 | 维持 |
| **50-79** | 🟡 关注 | 有一段时间未互动，或互动深度不足 | 建议 30 天内联系 |
| **20-49** | 🟠 预警 | 较长时间无实质性互动 | 建议尽快联系或重新评估强度 |
| **0-19** | 🔴 危险 | 极长时间无互动 | 建议考虑降级或标记为"需重新激活" |

#### 8.6.3 CLI 接口

```bash
# 查看所有联系人的健康分（排序输出）
python3 src/social.py health

# 查看单个人
python3 src/social.py health <contact>

# 只看需要关注的（健康分 < 50）
python3 src/social.py health --fix

# 排行榜
python3 src/social.py health --ranking

# 输出示例：
# 🟢 张总 92分 — 上周刚见过面，维持
# 🟡 李哥 54分 — 30天未联系，建议联系
# 🔴 王总 12分 — 180天未联系，建议降级评估
```

#### 8.6.4 仪表盘集成

仪表盘新增健康分板块：
- 健康分布饼图：🟢 X人 / 🟡 X人 / 🟠 X人 / 🔴 X人
- 平均健康分
- 健康分下降最快的 TOP5
- "需要你关注的健康问题" 列表（健康分低但有价值的联系人优先）

### 8.8 演进原则（v2.5.0 补充）

**原则六：信号先行。** 优先从已有数据中挖掘信息，再让用户补充——在要求用户输入之前，先问"系统已经知道什么"。

**原则七：保守推理，可追溯。** AI 推理结果必须标注置信度，低置信度的只追加不覆盖，所有自动变更可回溯。

**原则八：连接是关系的增强，不是替代。** 联系人之间的连接（connections）作为辅助信息存在，不用于替代强度计算或冷却检查，但可用于推荐"你可能也想联系××"。
