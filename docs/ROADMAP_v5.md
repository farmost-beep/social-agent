### 0.3 核心诊断

**诊断一：功能跑在了数据前面。**
系统已有目标锚定、建议引擎、兑现追踪三大支柱，但实际数据：34条timeline、0条outcome、10人有互动记录。advise引擎的"聊什么"在90%的core联系人上只能走规则降级（因为无timeline上下文）。**不是功能不够，是数据不够。**

**诊断二：锚定目标偏科。**
33个已锚定联系人全部指向"事业"维度，投资2人、知识6人、AI能力6人，家庭和健康为零。这反映了用户核心网络的职业属性，但也暴露了一个问题：**六维目标框架对"事业型人脉"过度拟合，对家庭/健康型关系缺乏锚定语言。** 一个家人的leverage怎么写？"撬动家庭目标"？框架需要适配非职业关系。

**诊断三：储备池不是真空，是休眠。**
4,365个reserve联系人全部有notes或tags（OCR导入），不是"灰色"而是"休眠"。标签分布显示强聚类：USTC(1334)、邮储(221)、职教社(110)、民建(106)。**这些不是需要补全的空白，而是等待激活的矿脉。** 信号驱动的晋升机制方向正确，但触发条件需要更积极。

**诊断四：timeline数据质量有隐患。**
34条timeline中14条的contact字段为None（41%），即互动记录未正确关联到联系人。这是数据采集质量问题，如果不修复，advise引擎和health评分都会失真。

## §1. 推演方法论

### 1.1 推演原则

本推演遵循产品已有的演进原则，并新增两条推演专用原则：

- **原则十三：数据先于功能。** 下一版优先解决数据密度问题，不新增依赖稀疏数据的功能。功能可以等数据，数据不能等功能。
- **原则十四：闭环优于开环。** 每个功能必须形成"输入→分析→输出→反馈→输入"闭环。开环功能（如outcome追踪，只有输入没有反馈）会被自然遗忘。

### 1.2 推演路径

```
v4.0.3 当前态
    │
    ├─ §2. v4.1 数据飞轮（解决诊断一+四）← 最紧迫
    │     └─ 目标：timeline 100条/月，数据质量 >90%
    │
    ├─ §3. v4.2 锚定深化（解决诊断二）← 并行推进
    │     └─ 目标：core层锚定 >80%，目标分布均衡
    │
    ├─ §4. v4.3 关系图谱解冻（解决诊断三的激活路径）← 依赖v4.1数据
    │     └─ 目标：500+推断连接，50+确认连接
    │
    ├─ §5. v5.0 社交战略层（从战术到战略）← 依赖v4.1-4.3
    │     └─ 目标：从"联系谁"到"如何经营整个网络"
    │
    ├─ §6. v5.1 多场景接入（降低输入摩擦）← 依赖v5.0
    │     └─ 目标：多通道被动+主动采集
    │
    └─ §7. v6.0 自主社交Agent（终极形态）← 长期愿景
          └─ 目标：从工具到Agent，从被动到主动
```

## §2. v4.1 数据飞轮（Data Flywheel）

> **代号：飞轮 —— 让数据自己转动起来**
> **优先级：P0，最紧迫**
> **依赖：无（可立即启动）**
> **预计周期：2-3周开发 + 持续运营**

### 2.1 核心问题

当前timeline仅34条/月，涉及10人。advise引擎的"聊什么"在90%的core联系人上走规则降级。**数据密度是所有高级功能的瓶颈。**

解法不是"让用户多记"（那是说教），而是**让记录的产出价值远大于输入成本**，形成正反馈飞轮：

```
记录互动 → timeline丰富 → advise更精准 → 用户觉得有用 → 更愿意记录
     ↑                                                    │
     └────────────────────────────────────────────────────┘
```

### 2.2 飞轮启动器：四个机制

#### 机制一：社交周报回溯（闭环反馈）

**现状**：`social advise` 只告诉你"该联系谁"，但不告诉你"上周你做了什么"。这是开环——只有输出没有反馈。

**方案**：新增 `social weekly` 命令，每周一推送不仅包含建议，还包含上周回溯：

```
📊 社交周报（2026-07-07 ~ 07-13）

【上周回溯】
- 记录互动 12 条（涉及 7 人）
- 完成待办 5 件，新增 3 件
- 最活跃：许封（4次互动）
- 冷却预警：3 人进入14天冷却

【本周建议】
1. 联系姜少 —— 21天未联系 + 锚定[事业]
   → social draft -c 姜少
2. ...

【飞轮指标】
- Timeline累计：34 → 46（+35%）
- Core层互动覆盖率：10/318 → 17/318（5.3%）
- 本周记录目标：15条（你上周做到了12条，很接近）
```

**关键设计**：
- "飞轮指标"让用户看到自己的数据积累进度，形成游戏化激励
- "本周记录目标"基于上周实际表现动态设定，不是硬指标
- 回溯让用户感受到"记录是有回报的"

#### 机制二：微信直连快速记录

**现状**：用户通过微信与Devin对话，但记录社交互动需要切换到CLI或Claude Code。摩擦虽小但存在。

**方案**：利用现有微信桥接（wechat-claude-code skill），用户在微信中直接说"记一下：和张总聊了项目进展"→ 自动路由到social-agent → 记录timeline → 微信回复"✅ 已记录"。

**技术路径**：
- 微信消息中包含"记一下"触发词 → social-agent skill自动激活
- 复用现有 `engine.add_timeline()` + `ai._ai_extract()` 提取待办
- 回复格式：`✅ 已记录：与张总聊项目进展 | 待办：跟进方案（P1）`
- 无需新增代码，只需在skill触发规则中确认"记一下"路由正确

#### 机制三：Timeline数据质量修复

**现状**：34条timeline中14条contact为None（41%数据质量问题）。

**方案**：
- `social timeline --fix` 命令：扫描contact为None的记录，尝试通过summary关键词匹配联系人
- `add_timeline()` 增加 contact 必填校验（空值拒绝写入）
- 匹配失败的记录标记 `unresolved: true`，在dashboard中提示

#### 机制四：重要日期补全

**现状**：0个important_dates。生日提醒功能完全空转。

**方案**：
- `social birthdays --scan`：从contacts的notes中自动提取生日模式（如"1129"→11月29日、"12月3日"→12-03）
- `social birthdays --add <contact> --date MM-DD`：手动添加
- 提取后自动生成年度提醒（每年提前7天推送）
- 这是一个**快速胜利**：notes中有大量生日信息（OCR导入），一次扫描可能补全数十个

### 2.3 成功指标

| 指标 | 当前 | v4.1目标 | 测量方式 |
|:----|:----:|:--------:|:--------|
| Timeline条数/月 | 34 | 100+ | 月末统计 |
| Timeline contact关联率 | 59% | >95% | `social timeline --stats` |
| Core层互动覆盖率 | 3.1% | >15% | 有timeline的core/total core |
| Important_dates | 0 | 50+ | `social birthdays --stats` |
| 周报推送阅读率 | — | >80% | 微信推送点击（需埋点） |

### 2.4 不做什么

- ❌ 不做自动聊天记录导入（合规风险，SPEC v4 §21.1已冻结）
- ❌ 不做语音记录（技术复杂度高，ROI不确定）
- ❌ 不做timeline的自动摘要生成（数据量太少，AI摘要价值低）

## §3. v4.2 锚定深化与目标框架适配

> **代号：深耕 —— 从33人到318人，从偏科到均衡**
> **优先级：P1，与v4.1并行**
> **依赖：v4.0 anchor功能（已完成）**

### 3.1 核心问题

33个已锚定联系人全部指向"事业"，家庭和健康为零。这不是用户没有家人朋友，而是**六维目标框架对非职业关系缺乏锚定语言**。

一个家人的leverage怎么写？
- `goals: ["家庭"]`
- `how: "子女教育的共同责任"`
- `direction: "互惠"`

这在技术上可行，但在语义上别扭——"撬动"这个词本身带有功利色彩，不适合家庭关系。**框架需要区分"撬动型关系"和"维系型关系"。**

### 3.2 目标框架升级

#### 3.2.1 关系类型二分

| 类型 | 特征 | 锚定语言 | 经营策略 |
|:----|:----|:--------|:--------|
| **撬动型** | 职业社交、信息交换、资源引荐 | leverage.how = "具体撬动方式" | 主动经营，定期联系 |
| **维系型** | 家人、挚友、健康伙伴 | leverage.how = "关系本身即是目的" | 自然互动，不需"经营" |

**数据模型变更**：
```json
"leverage": {
  "goals": ["家庭"],
  "how": "子女教育的共同责任",
  "direction": "互惠",
  "type": "维系",          // 新增：撬动 / 维系
  "confirmed": "2026-07-15"
}
```

`type: "维系"` 的联系人在advise引擎中不参与"冷却预警"——家人不需要14天冷却检查。

#### 3.2.2 锚定节奏调整

当前节奏：先31人（strength 4-5）→ 每批10人（strength 3）。

**调整建议**：
- 首批已完成33人（超额完成，原计划31人）
- 第二批：strength 3中tags含"民建"/"邮储"/"ustc"的聚类锚定（~80人，批量AI建议+抽样确认）
- 第三批：剩余strength 3联系人（~200人，AI建议+`--confirm`批量模式）
- **新增**：家人/健康类联系人的"维系型"锚定（不按strength排序，按关系类型优先）

### 3.3 聚类锚定

4,683个联系人的标签分布显示强聚类：USTC(1334)、邮储(221)、民建(106)、职教社(110)。

**方案**：对同一聚类内的core层联系人，AI建议可以共享上下文模板：
- "民建"标签的联系人 → leverage.goals大概率含"事业"，how大概率含"民建人脉+政策信息"
- "ustc"标签的联系人 → leverage.goals大概率含"知识"或"事业"，how大概率含"校友资源"

这不是批量设定（违反原则二），而是**AI建议的先验概率优化**——同一聚类内的建议更精准，用户确认效率更高。

### 3.4 成功指标

| 指标 | 当前 | v4.2目标 |
|:----|:----:|:--------:|
| Core层锚定率 | 10.4% | >80% |
| 目标分布均衡度 | 事业100% | 事业<60%，家庭/健康>10% |
| 维系型关系标注 | 0 | 20+ |
| 锚定效率（人/小时） | ~15 | >30（聚类加速） |

## §4. v4.3 关系图谱解冻

> **代号：织网 —— 从孤立点到连接图**
> **优先级：P2，依赖v4.1数据密度**
> **依赖：timeline数据 >100条，core锚定 >50%**

### 4.1 解冻条件评估

SPEC v4 §21.1 冻结 `_connections` 的理由是"零数据基础"。但实际情况是：
- 318个core联系人，标签聚类明显（USTC/邮储/民建/职教社）
- Timeline数据在v4.1后将达到100+条/月
- **连接不需要用户手动输入，可以从已有信号推断**

### 4.2 信号驱动连接推断

三层推断，置信度递减：

| 层级 | 信号 | 置信度 | 示例 |
|:----|:----|:------:|:----|
| **强** | 同tag + 同sub_relation + strength≥3 | 8-10 | 两个"ustc"+"金融同行"的core联系人 → 校友+同行双重连接 |
| **中** | 同tag + strength≥3 | 5-7 | 两个"民建"的core联系人 → 组织层连接 |
| **弱** | timeline共现（同一事件中提及两人） | 3-5 | "经张总引荐认识李总" → 张总-李总 引荐连接 |

**命令**：
```bash
social connect --infer           # 扫描全部core层，推断连接（仅写入置信度≥5的）
social connect --infer --dry-run # 预览推断结果
social connect A B --type 同事    # 手动添加
social connections 张三            # 查看张三的关系网
social find-path 张三 李四        # 路径查找
social graph --contact 张三       # ego网络可视化
```

### 4.3 引荐路径建议

当advise引擎建议"联系X"时，如果X与用户之间有冷却但X与某个活跃联系人Y有连接，可以建议：

```
建议联系李总（21天未联系 + 锚定[事业]）
💡 路径：你 → 王总（上周刚联系）→ 李总（同事）
   可请王总引荐，顺势恢复联系
```

这是advise引擎的**自然升级**，不需要新的数据采集——只需要connections图谱 + timeline + advise三者交叉。

### 4.4 成功指标

| 指标 | 目标 |
|:----|:----:|
| 推断连接数 | 500+ |
| 确认连接数 | 50+ |
| find-path平均跳数 | <3 |
| 引荐路径建议命中率 | >30%（建议后被采纳） |

## §5. v5.0 社交战略层

> **代号：军师 —— 从"联系谁"到"如何经营整个网络"**
> **优先级：P3，依赖v4.1-4.3全部完成**
> **这是主版本升级，从温度计→参谋→军师**

### 5.1 战略层定义

v4.0的advise引擎是**战术级**的：告诉你这周联系谁、为什么、聊什么。
v5.0要升级到**战略级**的：告诉你如何经营整个关系网络，如何分配社交资本，如何在不同阶段采取不同策略。

### 5.2 四个战略能力

#### 5.2.1 社交组合分析

**类比**：像投资组合分析一样分析关系组合。

```bash
social portfolio
```

输出：
```
📊 社交组合分析（2026 Q3）

【目标分布】
- 事业: 187人 (58.8%) ████████████████
- 投资: 32人 (10.1%) ███
- 知识: 45人 (14.2%) ████
- AI能力: 28人 (8.8%) ██
- 家庭: 18人 (5.7%) █
- 健康: 8人 (2.5%) ▌

⚠️ 过度集中：事业圈占58.8%，家庭+健康仅8.2%
   建议：Q3重点补充健康圈人脉（健身伙伴/医生/营养师）

【互动分布】
- 活跃（30天内有互动）: 23人 (7.2%)
- 温和（30-90天）: 45人 (14.2%)
- 冷却（90-180天）: 120人 (37.7%)
- 休眠（>180天）: 130人 (40.9%)

⚠️ 40.9%的core联系人处于休眠状态
   建议：启动"激活战役"，每月激活5个休眠关系
```

**关键设计**：
- 不做ROI计算（原则七），但做**分布诊断**
- 诊断结果直接关联到advise引擎的优先级调整
- "激活战役"是可执行的，不是空泛建议

#### 5.2.2 关系生命周期管理

每段关系有生命周期阶段：

| 阶段 | 特征 | 经营策略 | 系统行为 |
|:----|:----|:--------|:--------|
| **新识** | timeline≤2条，strength≤2 | 深化了解，寻找交集 | 建议添加memories、识别共同标签 |
| **深化** | timeline 3-10条，strength 3 | 增加互动频率，探索合作 | 建议每月1-2次互动 |
| **成熟** | timeline>10条，strength 4-5 | 稳定维护，兑现价值 | 常规冷却检查 + outcome追踪 |
| **维护** | 成熟但互动频率下降 | 保持温度，低成本维护 | 14天预警，建议轻量互动（朋友圈点赞/节日祝福） |
| **休眠** | >180天无互动 | 评估是否值得激活 | 建议降级评估 或 激活战役 |
| **激活** | 从休眠中恢复 | 重新建立连接 | 建议面对面（最高depth互动） |

**命令**：
```bash
social lifecycle               # 全体core层生命周期分布
social lifecycle 张三          # 单人生命周期诊断
social lifecycle --stage 休眠  # 列出所有休眠关系
```

#### 5.2.3 季节性策略

不同目标维度在不同时间窗口的重要性不同：

| 季度 | 重点目标 | 社交策略 |
|:----|:----|:--------|
| Q1 | 事业+知识 | 年初规划期，重点经营事业导师和知识圈 |
| Q2 | 投资+AI能力 | 业绩窗口期，投资圈信息交换密集 |
| Q3 | 事业+家庭 | 下半年冲刺 + 暑期家庭活动 |
| Q4 | 投资+家庭 | 年终总结 + 基金排榜 + 节日家庭维护 |

系统根据季节自动调整advise引擎的信号权重：
- Q2的advise中，投资圈联系人的冷却阈值从14天缩短到10天
- Q4的advise中，家庭圈联系人优先级提升

#### 5.2.4 社交ROI回溯（保守版）

当outcome数据积累到50+条（预计v4.1后6个月），系统可以生成半年度社交ROI回溯：

```bash
social roi --half 2026H2
```

输出：
```
📈 2026下半年社交投入产出回溯

【投入】
- 记录互动 312 次（涉及 45 人）
- 投入时间估算：~52小时（按每次互动10分钟估）
- 最多投入：许封（28次/4.7h）、姜少（22次/3.7h）

【产出】（outcome记录）
- 事业成果：5条（含2次引荐、1次合作机会、2次关键信息）
- 投资成果：3条（含1次投资线索、2次市场判断验证）
- 知识成果：2条（含1次方法论交流、1次书单推荐）

【高ROI关系】
- 张总：3次互动 → 2条outcome（引荐+信息）
- 王总：5次互动 → 1条outcome（合作机会）

【低ROI关系】
- 李总：12次互动 → 0条outcome
  诊断：高频互动但未产生可记录成果，建议评估互动深度
```

- 只记录可追溯事实，不做货币化量化
- 明确标注"不代表全部价值"
- 低ROI不是降级建议，而是"评估互动深度"的诊断提示

## §6. v5.1 多场景接入

> **代号：渗透 —— 让系统无处不在但从不打扰**
> **优先级：P4，依赖v5.0战略层**
> **目标：从"主动记录"到"被动+主动"混合采集**

### 6.1 日历集成

**场景**：用户日历中有"和张总午餐"事件 → 系统自动在事件前1小时推送"你和张总上次聊到XX项目，今天可以跟进" → 事件后自动提示"记一下午餐要点"。

**技术路径**：
- 读取 macOS Calendar 数据库（SQLite，本地无API依赖）
- 匹配日历事件标题/参与人与contacts
- 事件前推送（复用remind机制）+ 事件后提示记录

### 6.2 邮件签名解析

**场景**：用户收到新邮件 → 系统扫描签名 → 发现新联系人 → 提示"检测到新联系人：李四，XX公司VP，是否添加？"

**技术路径**：
- macOS Mail.app AppleScript 提取最近邮件签名
- 正则解析姓名/公司/职位/电话/邮箱
- 与现有contacts去重后提示添加
- **只扫描，不自动添加**（遵守原则二）

### 6.3 社交媒体变化感知

**场景**：某联系人在LinkedIn上换了工作 → 系统检测到变化 → 提示"张总从A公司跳到B公司了，要不要发个消息祝贺？"

**技术路径**：
- 定期（月度）通过DuckDuckGo搜索core层联系人的公开信息变化
- 对比上次enrich结果，检测变化
- 变化触发advise引擎的"重要事件"信号

### 6.4 家庭共享模式

**场景**：夫妻共同管理某些家庭关系（孩子的老师、父母的主治医生）。

**数据模型**：
```json
{
  "shared_with": ["spouse@example.com"],
  "shared_scope": ["家庭"]
}
```

**约束**：
- 只共享"家庭"目标维度的联系人
- 共享是只读（对方可以看，不能改）
- 需要双方都安装social-agent

### 6.5 不做什么

- ❌ 不做微信聊天记录自动导入（合规风险，永久冻结）
- ❌ 不做通话记录自动导入（同上）
- ❌ 不做GPS位置社交建议（隐私风险，且"附近的人"是微信的功能不是CRM的功能）
- ❌ 不做社交媒体自动发帖/自动点赞（社交关系经营不可自动化，这是产品的哲学底线）

## §7. v6.0 自主社交Agent

> **代号：觉醒 —— 从工具到Agent**
> **优先级：长期愿景，2027下半年+**
> **依赖：v5.0+全部完成，数据积累12个月+**

### 7.1 终极形态定义

v6.0的social-agent不再是"用户主动查询才回答"的工具，而是**持续感知、主动建议、情境驱动**的Agent。

| 维度 | v4.0（工具） | v6.0（Agent） |
|:----|:----|:----|
| 触发方式 | 用户主动查询/定时推送 | 情境感知+实时建议 |
| 建议频率 | 每周固定 | 按需触发（事件驱动） |
| 数据来源 | 用户手动记录 | 多通道被动+主动 |
| 分析深度 | 单联系人诊断 | 全网络战略分析 |
| 主动性 | 被动等待用户输入 | 主动识别机会和风险 |

### 7.2 三个Agent能力

#### 7.2.1 情境感知引擎

**场景示例**：
- 用户日历显示"14:00 与张总会议" → 11:00 推送："张总上次聊到XX项目，今天可以问进展。他的公司最近有融资新闻，可能心情不错。"
- 用户刚在微信中和李总结束对话 → 自动提示："要不要记一下和李总的对话要点？"
- 用户出差到北京 → 推送："你在北京有7个core联系人，要不要约个咖啡？"

**技术基础**：
- 日历集成（v5.1）
- 位置感知（macOS Core Location，需用户授权）
- 微信消息事件感知（通过桥接，仅感知消息事件不读取内容）

#### 7.2.2 社交知识图谱

将所有数据融合为统一知识图谱：

```
联系人 ──连接──→ 联系人
  │                │
  ├─timeline─→ 事件 ──共现──→ 事件
  │
  ├─leverage─→ 目标
  │
  ├─outcome──→ 成果 ──关联──→ 目标
  │
  └─tag──────→ 聚类 ──包含──→ 联系人
```

**查询语言**（`social query`）：
```
social query "找出所有事业锚定、但90天未联系、且有pending待办的联系人"
social query "我的投资圈中，谁认识XX基金的决策人？"
social query "过去半年，哪些关系产生了outcome？"
social query "我的社交网络中，最大的桥接节点是谁？"
```

#### 7.2.3 年度社交回顾

```bash
social annual 2026
```

生成年度社交回顾报告（PDF），包含：
- 年度互动统计（总互动次数、涉及人数、最活跃关系）
- 目标维度分布变化（年初 vs 年末）
- 新增核心关系 / 关系深化 / 关系降级
- Outcome汇总（按目标维度）
- 社交网络增长（新连接、网络密度变化）
- 来年建议（基于趋势外推）

## §8. 风险与约束

### 8.1 数据隐私风险

随着数据密度增加，contacts.json + timeline.json + todos.json 成为高价值隐私数据。

**建议**：
- v4.1起新增 `social backup --encrypt`：AES加密备份到指定路径
- v5.0起新增 `social export --redact`：导出时脱敏（姓名→哈希、电话→掩码）
- 本地存储不上云的原则不变

### 8.2 社交异化风险

系统可能将人际关系变成"任务清单"，导致用户与朋友交流时心里想着"这算一次互动记录"。

**缓解**：
- advise条数封顶3-5条/周（已有，保持）
- 周报中不显示"互动次数排名"（避免将朋友量化比较）
- 维系型关系不参与冷却检查（§3.2.1）
- 在SKILL.md中新增原则：**关系经营是手段不是目的，系统服务于关系本身，而非替代关系**

### 8.3 范围蔓延风险

本推演列出了v4.1到v6.0共6个版本的功能。如果不加约束，每个版本都会膨胀。

**约束**：
- 每个版本只做该版本的核心功能，不做"顺便加的"
- 冻结功能（connections/groups/import）的解冻必须满足明确条件
- 新功能必须有成功指标，达不到指标的降级或回退

### 8.4 单用户架构极限

当前架构是单用户单机本地存储。如果数据量增长到10,000+联系人、5,000+timeline、500+connections，JSON文件读写性能会下降。

**预判**：
- v5.0前不需要处理（4,683联系人的JSON读写仍在毫秒级）
- v5.0时评估是否迁移到SQLite（保持本地，不上云）
- v6.0时如果引入家庭共享，需要多用户数据同步机制
